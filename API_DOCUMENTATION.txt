2) AUTHENTICATION METHOD
==================================================
Token strategy:
1. Access Token (JWT)
   - Created by createAccessToken() in src/Utils/token.js
   - Secret: ACCESS_TOKEN_SECRET
   - Expiry: 15m
   - Expected in header: Authorization: Bearer <token>

2. Refresh Token (JWT)
   - Created by createRefreshToken() in src/Utils/token.js
   - Secret: REFRESH_TOKEN_SECRET
   - Expiry: 7d
   - Sent in httpOnly cookie: refreshToken
   - Cookie options in login:
     - httpOnly: true
     - secure: true
     - sameSite: strict
     - maxAge: 7 days

Refresh flow:
- POST /auth/refresh reads req.cookies.refreshToken, verifies token, returns new accessToken.

Logout flow:
- POST /auth/logout clears refresh token cookie and nulls saved refresh token in User collection.

IMPORTANT IMPLEMENTATION NOTES:
- Most routes are not protected by auth middleware (`auth.middleware.js` is mostly unused).
- `/employees/:employeeId` delete reads `req.user.id` but route does not apply auth middleware, so this may fail.
- Refresh service checks User collection only; employee refresh-token persistence differs from user persistence.

5.1 AUTH ROUTES (/auth)
File: src/UserAuth/UserAuth.route.js
----------------------------------------------

POST /auth/signup
- Multipart/form-data (optional image file under "image")
- Body (typical): name, email, password, role, ...profile fields
- Creates user in User collection with hashed password
- Response: 201 { message, user: { id, email } }

POST /auth/login
- Body: { email, password, role? } (role default "user" in controller)
- Logs in company user or employee (see service logic)
- Sets refreshToken cookie
- Response: { accessToken, user: { id, name, email, profileImage, role } }

POST /auth/superadmin-login
- Body: { email, password, role? } (role defaults "superadmin")
- Superadmin login endpoint
- Sets refreshToken cookie
- Response: { accessToken, user: { id, name, email, phone, profileImage, role } }

POST /auth/login-company
- Body: { email, password }
- Forces role = "admin"
- Sets refreshToken cookie
- Response: { accessToken, user: { id, name, email, role, profileImage } }

POST /auth/refresh
- Reads refreshToken cookie
- Response: { accessToken, user: { id, email, profileImage, role } }
- 403 on failure

POST /auth/logout
- Reads refreshToken cookie
- Clears cookie and removes token reference
- Response: { message: "Logged out" }

POST /auth/forgot-password
- Body: { email }
- Generates and stores OTP + expiry in User
- Sends email via SMTP
- Response: { message: "OTP sent to your email/phone" }

POST /auth/verify-otp
- Body: { email, otp }
- Validates OTP and expiry
- Response: { message: "OTP verified" }

POST /auth/set-new-password
- Body: { email, newPassword }
- Sets new hashed password (min length 8 check in service)
- Response: { message: "Password updated successfully" }

POST /auth/admin/set-new-password
- Body: { email, newPassword }
- Same as above but for role superadmin

PATCH /auth/change-password
- Body: { userId, currentPassword, newPassword }
- Verifies current password then updates password hash

GET /auth/:userId
- Gets one user by ID

PATCH /auth/:userId
- Multipart/form-data (optional "image")
- Updates user profile fields

GET /auth/totalEmployeeProject/:userId
- Returns totalProjects + totalEmployees for a company/user

ROUTE ORDER NOTE:
- In route file, GET /:userId appears before GET /totalEmployeeProject/:userId.
- Because of this order, /auth/totalEmployeeProject/:userId can be captured by /auth/:userId and fail.
